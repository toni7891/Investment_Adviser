  Slide 2 — The Problem

  Three real problems this project solves:

  1. AI at scale — Running AI models locally requires expensive GPU hardware, constant maintenance, and manual updates. How do we get AI insights without
  managing infrastructure?
  2. Secrets security — Storing database passwords and API keys in .env files on a server is dangerous. If the server is compromised, all secrets are exposed.
  3. Credential management — How does an EC2 instance call AWS services (Bedrock) without hardcoding AWS access keys in the code?

  ---
  Slide 3 — Service Overview

  AWS Bedrock

  - Fully managed AI service — access foundation models (Claude, Amazon Nova, Titan) via a single API
  - No GPU management, no model training, no infrastructure
  - Supports a Converse API — one interface works across all model families
  - Pay per token (very cheap for small apps)

  AWS SSM Parameter Store

  - Secure, managed storage for secrets and configuration
  - SecureString type encrypts values using AWS KMS
  - App calls SSM at startup — secrets never touch disk
  - Free tier: first 10,000 standard parameters free

  IAM Roles for EC2

  - Instead of putting AWS credentials in files, you attach a Role to the EC2 instance
  - boto3 (AWS SDK) automatically discovers the role — zero credentials in code
  - Principle of least privilege: role only has the exact permissions needed

  Slide 5 — Deployment Steps

  What was created in AWS:

  1. IAM Role (investment-adviser-forec2-bedrock)
    - Attached policies: AmazonBedrockFullAccess, AmazonSSMReadOnlyAccess
    - Role attached directly to EC2 instance
  2. SSM Parameter Store — 5 parameters created:
  | Parameter                          | Type         | Value                     |
  |------------------------------------|--------------|---------------------------|
  | /investment-manager/MONGO_URI      | SecureString | MongoDB connection string |
  | /investment-manager/JWT_SECRET_KEY | SecureString | JWT signing key           |
  | /investment-manager/LLM_BACKEND    | String       | bedrock                   |
  | /investment-manager/BEDROCK_MODEL  | String       | amazon.nova-micro-v1:0    |
  | /investment-manager/BEDROCK_REGION | String       | us-east-1                 |

  3. EC2 Instance — t3.micro, Ubuntu 22.04, IAM Role attached, no .env file on disk
  4. Bedrock Model Access — Requested and approved in Bedrock console (us-east-1)

  Key code — SSM loading at startup:
  def _load_ssm_secrets():
      ssm = boto3.client("ssm", region_name="us-east-1")
      params = ssm.get_parameters_by_path(
          Path="/investment-manager/", WithDecryption=True
      )
      for p in params["Parameters"]:
          key = p["Name"].split("/")[-1]
          os.environ.setdefault(key, p["Value"])

  Key code — Bedrock Converse API call:
  client = boto3.client("bedrock-runtime", region_name="us-east-1")
  response = client.converse(
      modelId="amazon.nova-micro-v1:0",
      messages=[{"role": "user", "content": [{"text": prompt}]}],
      inferenceConfig={"maxTokens": 2048},
  )

  - Enable Bedrock Guardrails to filter harmful AI outputs

  - Add WAF in front of the application to block malicious requests
  - Use Auto Scaling Group + Load Balancer for high availability

  Step 16 — CloudWatch Logs: How it works technically

  The problem: App logs only existed on the EC2 server's journald (local disk). No visibility without SSH.

  What we did:

  1. Added watchtower library to requirements.txt — this is a Python logging handler that knows how to talk to CloudWatch.
  2. Added _setup_cloudwatch_logging() to main.py — at app startup, this function:
    - Reads CLOUDWATCH_LOG_GROUP from the environment (loaded from SSM)
    - Creates a boto3 CloudWatch Logs client (authenticates via the EC2 IAM Role — no keys needed)
    - Attaches a CloudWatchLogHandler to Python's root logger
  3. IAM Role — added CloudWatchLogsFullAccess to the EC2 role so the server has permission to write logs to CloudWatch.
  4. SSM Parameter — stored /investment-manager/CLOUDWATCH_LOG_GROUP = /investment-manager/app so the log group name is configurable without changing code.

  The flow:
  FastAPI logger.info() → Python root logger → watchtower handler → boto3 → CloudWatch Logs API → /investment-manager/app/fastapi stream

  Step 17 — SES Email Alerts: How it works technically
  
  The problem: No way to know when your portfolio makes a big move unless you're actively watching the dashboard.

  What we did:

  1. Added 3 SSM parameters — ALERT_ENABLED, ALERT_EMAIL, ALERT_THRESHOLD_PCT — all configurable without code changes.
  2. Added _maybe_send_alert() to routes.py — a helper function that:
    - Checks if alerts are enabled and a threshold is configured
    - Compares the portfolio's daily_change_pct against the threshold
    - Uses an in-memory dict _ALERT_SENT as a cooldown (one alert per portfolio per day)
    - Calls ses.send_email() via boto3 if the threshold is exceeded
  3. Hooked it into the existing portfolio GET endpoint — every time someone (or the Lambda) loads a portfolio, _maybe_send_alert() runs automatically. No new
  API route, no separate process.
  4. IAM Role — added ses:SendEmail inline policy so the EC2 instance has permission to send emails.
  5. SES verified identity — AWS requires you to verify the sender email address before it lets you send from it.

  The flow:
  GET /api/portfolios/4RCH3R
    → yfinance fetches live prices
    → daily_change_pct computed
    → _maybe_send_alert() checks threshold
    → boto3 SES → email to toni7891@gmail.com

  ---
  Key architectural point for the presentation: Both features authenticate to AWS using the EC2 IAM Role — no API keys, no credentials stored anywhere on the
  server. boto3 automatically picks up the role from the EC2 instance metadata. This is the same pattern used for Bedrock and SSM.
